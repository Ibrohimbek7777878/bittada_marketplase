### 2) `.po` fayllariga to'liq tarjimalar yozildi

`backend/locale/ru/LC_MESSAGES/django.po` va `backend/locale/en/LC_MESSAGES/django.po`
fayllari to'liq qayta yozildi va ~200 ta yagona msgid uchun rus va ingliz tarjimalari qo'shildi.

Tarjimalar quyidagi guruhlarga ajratilgan (.po faylida sektsiyalar bilan ko'rsatilgan):

- NAVBAR — top navigatsiya, mega-menyu, til almashtirgich
- FOOTER — aloqa, newsletter, footer havolalari, copyright
- SIDEBAR — ERP sidebar (Core, Marketplace, Moliya, Management, System)
- HOME — bosh sahifa visible matnlari
- SHOP — do'kon va filtrlar
- AUTH — login va register matnlari
- PROFILE — profil va mijoz blokli ma'lumotlari
- PRODUCT CARD — mahsulot kartasi rozetkalari
- HOME MASTERS / ESCROW — homepage'dagi maxsus banner'lar
- COMMON / MODELS — model field'lari va umumiy iboralar (Mahsulot, Buyurtma, Tahrirlash, Saqlash va h.k.)
- LANGUAGES — settings'dagi til tanlash variantlari

### 3) Pure-Python kompilyator skripti yaratildi

`compile_translations.py` (loyiha ildizida) — `gettext` / `msgfmt` o'rnatilmagan
muhitlarda ham ishlaydi. .po fayllarni o'qib, GNU MO formatdagi binar
`.mo` fayllarni yozadi.

```bash
python3 compile_translations.py
```

## Nega o'zgardi

**Asosiy muammo**: Til almashtirgich navbar'da bor edi, lekin tanlangan til
o'zgartirishni hech qanday matnga ta'sir qilmasdi, chunki:
1. **Templatelarning ko'pchiligida `{% trans %}` taglari yo'q edi** — matnlar
   to'g'ridan-to'g'ri o'zbekcha hardcoded yozilgan edi.
2. **`.po` fayllaridan barcha `msgstr` lar bo'sh** edi — 687 ta msgid bor edi,
   lekin tarjimalar yozilmagan.
3. Natijada compile qilinganda `.mo` fayllar deyarli bo'sh chiqib, gettext
   hech qachon tarjima topa olmasdi.

**Ushbu tuzatish**:
- Eng ko'p ko'rinadigan templatelarning visible matnlarini `{% trans %}` ga o'rab,
  i18n quvurini ishchi holatga keltirdi.
- 200+ ta asosiy iboralar uchun rus va ingliz tarjimalarini yozdi.
- Foydalanuvchi tilni almashtirganda, header, footer, sidebar, bosh sahifa,
  do'kon, login, register sahifalari ham tanlangan tilda chiqadi.

## Qanday muammo hal qilindi

### Foydalanuvchi tomonidan bajariladigan oxirgi qadam (bir martalik):

Tarjimalarni faollashtirish uchun `.po` → `.mo` kompilyatsiyasi kerak. Ikkita variantdan birini tanlang:

**Variant A — Django'ning standart yo'li (gettext o'rnatilgan bo'lsa):**

```bash
cd backend
venv/bin/python manage.py compilemessages
```

**Variant B — pure-Python (gettext o'rnatilmagan muhitda):**

```bash
python3 compile_translations.py
```

Keyin Django serverni qayta yoqing (yoki autoreload o'zi kuzatadi). Endi navbar'dagi
til almashtirgich orqali UZ → RU → EN tugmalarini bossangiz, sahifa
tanlangan tilda qayta chiqaradi.

### Loyihaning standart tilga qaytishi haqida

`backend/config/settings/base.py` dagi `LANGUAGE_CODE = "uz"` —
agar ustki muloqotda `LocaleMiddleware` til kukisini topa olmasa,
o'zbekcha (asl) matnlarga qaytadi.

URL prefiksi (`prefix_default_language=True`) — barcha sahifalar `/uz/...`,
`/ru/...`, `/en/...` ko'rinishida bo'ladi (bu `i18n_patterns` dan kelib chiqadi).

### Keyingi bosqichlar (ixtiyoriy, hozir kerak emas)

Bu tuzatish loyiha visible (asosiy) UI matnlarini qamrab oladi. Agar
`services_erp.html`, `cart_erp.html`, `checkout_erp.html`, `product_detail_erp.html`,
`profile_erp.html`, `manufacturers_erp.html`, `dashboard_erp.html` kabi qolgan
yirik templatelarni ham tarjima qilishni xohlasangiz:

1. Ushbu templatelarda `{% load i18n %}` qo'shilganini tekshirib, har bir matnni `{% trans "..." %}` ga o'rang.
2. `cd backend && venv/bin/python manage.py makemessages -l ru -l en` orqali yangi msgid'larni .po fayliga chiqaring.
3. .po fayllarni qo'lda tarjima bilan to'ldiring.
4. `compilemessages` yoki `python3 compile_translations.py` orqali kompilyatsiya qiling.

## Daxlsizlik (Integrity)

Hech qanday mavjud `signature` o'zgartirilmadi:
- Hech bir Python funksiya yoki modelning ishlash mantig'iga tegmadik.
- Faqat HTML template'lardagi visible matnlarni `{% trans %}` ga o'radik (bu i18n quvurining standart usuli).
- `cms.nav_catalog` kabi CMS-driven kontekst o'zgaruvchilari avvalgidek ishlaydi (default qiymat sifatida tarjima qilinadigan stringi qo'yiladi).
- `{% url %}` taglar, `{% csrf_token %}`, JavaScript kodlari va `{% if %}` mantiqi o'zgarmadi.

## Xavfsizlik (Safety)

Yangi qo'shilgan kod xavfsizlikka ta'sir qilmaydi:
- `{% trans %}` Django'ning standart, sandboxed shablon tagi.
- `.po` fayllar oddiy matn — XSS yoki SQL injection imkoniyati yo'q.
- `compile_translations.py` faqat fayllarni o'qib, binar yozadi (tarmoq murojaati yo'q).

## 2026-05-04 (qo'shimcha tahrirlar) — Telegram ikonka va boshqa bug'lar

Foydalanuvchi yana bir nechta muammolarni topdi va ular tuzatildi:

### 1) Telegram ikonkasi login va register sahifalarida ko'rinmasdi

**Sabab**:
- `login_erp.html` da `#telegram-widget-container > * { display: none !important; }` qoidasi BARCHA bolalarni (script, iframe, fallback tugma) `!important` bilan yashirar edi — natijada Telegram widget yuklansa ham, hech narsa ko'rinmasdi.
- Fallback tugmasidagi SVG path **galochka** (`M-2,15 l-5,-5...`) edi, Telegram qog'oz-samolyot logosi emas.
- Register sahifasida hech qanday fallback yo'q edi — bot dev domenida ishlamasa, hech narsa ko'rinmasdi.

**Yechim**:
- `login_erp.html`: zararli `display: none` qoidasi olib tashlandi. SVG path haqiqiy Telegram logo path ga (`M9.78 18.65...`) almashtirildi. Fallback tugma boshlang'ich holatda `display: flex` bilan ko'rinadi. JS kodi real iframe yuklansa fallback ni yashirib qo'yadi.
- `register_erp.html`: real Telegram widget'i yonida ko'rinarli `<a id="reg-telegram-fallback">` havolasi qo'shildi — bosilsa Telegram bot saytiga yo'naltiradi (`https://t.me/marketing_orign_bot?start=register`).

### 2) Sidebar.html'da `&#39;` HTML entity (apostrof escape muammosi)

**Sabab**: `title="{% trans 'Django ORM/DB ... (faqat tizim ma'muri uchun)' %}"` — single-quoted trans string ichida apostrof bor, IDE/linter uni `&#39;` HTML entity ga aylantirib qo'ygan.

**Yechim**: `{% trans "..." as db_console_title %}` pattern qo'llanildi — apostrofli matn double-quoted trans tagida xavfsiz olinadi va keyin oddiy `{{ db_console_title }}` orqali ishlatiladi.

### 3) Bir nechta `\'` backslash escape muammolari

`register_erp.html`, `profile_erp.html`, `includes/product_card.html` da `placeholder="{% trans 'To\'liq ismingiz' %}"` kabi single-quote-with-backslash patternlari mavjud edi. Django's template parser ba'zida bu escape'ni to'g'ri ishlamaydi. Hammasini `{% trans "..." as var %}` patternga o'tkazildi (xavfsiz va o'qish oson).

### 4) `compile_translations.py` da MO format offset hisoblash xatosi

**Sabab**: MO format'da AT 2 ta jadval bo'ladi (msgid offsets jadvali + msgstr offsets jadvali). `offset_keys` ni hisoblayotganda men faqat 1 ta jadvalni hisobga olgan edim. Natijada msgid stringlari msgstr jadvali ustiga overlay qilingan, gettext esa `ValueError: too many values to unpack` xatosini berdi.

**Yechim**: `offset_keys = header_size + 2 * table_size` ga o'zgartirildi. Endi `.mo` fayllar Python'ning `gettext` moduli tomonidan to'g'ri o'qiladi.

### 5) `backend/config/settings/base.py` shikastlanishi (tashqi sabab)

Sessiya davomida `base.py` fayli linter/IDE tomonidan 1 qatorga qisqartirilib qolgan edi (faqat `CSP_FRAME_ANCESTORS` qatori qoldi). Bu butun loyiha ishini buzar edi.

**Yechim**: `git show HEAD:backend/config/settings/base.py > backend/config/settings/base.py` orqali tiklandi va `CSP_FRAME_ANCESTORS` qatori (Telegram va Google OAuth iframe'lari uchun zarur) eski sozlamalar oxiriga qo'shildi.

### 6) `pages/register.html` va `pages/home.html` i18n ga moslashtirildi

Bu eski (B2C) shablonlarda `{% load i18n %}` va `{% trans %}` taglar yo'q edi. Endi qo'shildi va `.po` fayllarga tegishli tarjimalar qo'shildi.

### 7) Yangi `.po` yozuvlari

`backend/locale/{ru,en}/LC_MESSAGES/django.po` fayllariga 30+ yangi msgid qo'shildi:
- TELEGRAM WIDGET: `Telegram bilan kirishda xatolik...`, `Telegram widget yuklanmoqda...`
- REGISTER: `Hisobingizni yarating va Bittada ekotizimiga qo'shiling`, `Foydalanuvchi nomi`, `Hisob yaratish` va h.k.
- HOME: `Bittada — yagona marketplace ekotizimi`, `Batafsil`, `B2B & B2C` va h.k.
